# ■ スクラム道場：進行用マニュアル（講義者用）

## 1. 開発サイクル

### 全体の流れ

| ステップ | 時間 | 内容 |
| :--- | :--- | :--- |
| 導入 | 15分 | 今日は何をやるのかの概要の説明をする |
| 編成 | 10分 | 3-5人程度のグループに分け、ロールの設定を行う |
| 実践 | 60*n分 | スプリントを一回回す |
| まとめ | 15分 | 振り返り、成果の共有、エンジニアリングの「考え方」の教訓 |

### プロジェクト導入コンセプト：『教室』から『開発現場』へのパラダイムシフト

子供たちに最初に伝えるべきは、ここが「先生に教わる場所」ではなく、**「君たちが決定権を握り、世界へ実装する場所」**であるという事実です。

#### 1. 「オーナー」という役割の定義（責任と権限）
*   **メッセージ:** 「このプロジェクトの主役は、君たちです」
*   **本質:** 先生はコードを書く道具（実装エンジニア）であり、何をどう作り、どう改善するかを決める権利（プロダクトオーナー）はすべて子供にあることを強調する。
*   **ワクワク感:** 「自分の判断でゲームが変わる」という、クリエイターとしての圧倒的な自由を提示する。

#### 2. 「プロセス」という最強の武器（体験価値）
*   **メッセージ:** 「完成したゲームより、そこに至る『思考の跡』の方が価値がある」
*   **本質:** ゲームが動くことは結果に過ぎない。壁にぶつかり、AIやチームと対話し、試行錯誤して解決したプロセスこそが、彼らが将来どんな課題にも立ち向かえる「自分だけの攻略本（資産）」になることを説く。
*   **ワクワク感:** 「自分たちだけのオリジナルの武器（技術知見）を構築していく」という成長の物語を感じさせる。

#### 3. 「プロの現場」という臨場感（エンジニアリングの体験）
*   **メッセージ:** 「遊びではなく、実戦を体験する」
*   **本質:** VS Codeでのコーディング、GitHubでの管理、ブラウザへのデプロイというフローは、プロの現場そのもの。彼らが作るゲームは、インターネットを通じて世界中の誰でも遊べる「本物のプロダクト」であると認識させる。
*   **ワクワク感:** 「自分のPCから世界に向けて何かを発信する」という、圧倒的な当事者意識と公開の喜び。

#### この導入が目指す到達点
この説明を聞いた後、子供たちが以下の状態になることを意図しています。

*   **「失敗してもいい、それはプロセス（資産）だから」**という安心感。
*   **「次はどうしようか？」と自ら考え始める**能動的な姿勢。
*   **「自分はゲームを作る側の人間なんだ」**というアイデンティティの変容。


### スプリント内容（1スプリント：60分）

| フェーズ | 時間 | 内容 |
| :--- | :--- | :--- |
| **1. 計画** | 10分 | ゲームをプレイし、リーダーと「次の最優先改善」を決定する |
| **2. 設計** | 20分 | 各担当の「動き」をホワイトボードに図解させる（SMは見守る） |
| **3. 調整** | 10分 | 設計図の矛盾を指摘し、チーム全員で解決策を握る |
| **4. 統合** | 10分 | 子供たちの仕様をマージし、システムとして動かす |
| **5. 振り返り**| 10分 | 動かした感想から、次回の「やりたいこと」を蓄積する |

## 2. SM（あなた）の鉄則
* **独裁の防波堤:** 「キャプテンはどう思う？」「メンバーはどう思う？」と問いかけ、全員を議論に巻き込む
* **失敗の再定義:** バグやエラーは「失敗」ではなく「設計の矛盾（データ）」と言い換える
* **チーム主体:** 「俺がやる」のではなく、彼らが考えた仕様が形になる「パートナー」として振る舞う

## 3. 開始前チェックリスト
- [ ] リーダー（キャプテン）は決定しているか？
- [ ] ホワイトボードとペン（設計図用）はあるか？
- [ ] 「やりたいことリスト（バックログ）」は見える位置にあるか？
- [ ] 開発環境（AI・PC）はすぐ使えるか？
